## 一、项目背景

### 1.1 业务现状

视频剪辑软件作为创作者工具，用户规模持续增长。但在数据管理方面存在以下问题：

问题1：数据口径不统一
- 产品、运营、财务各自定义指标
- 同一个指标不同人算出来结果不同
- 周会经常因为数据对不上而争论

问题2：数据获取效率低
- 临时数据需求依赖数据分析师手动处理
- 从提需求到拿到数据平均需要3-5天
- 分析师疲于应付取数，无暇做深度分析

问题3：缺乏统一看板
- 各部门使用不同的Excel和BI工具
- 数据分散，难以形成全局视角
- 管理层无法快速了解业务全貌

问题4：AI能力未充分利用
- 数据分析仍以人工为主
- 异常发现依赖人工巡检
- 缺乏智能化的分析手段

### 1.2 项目目标

建设数据管理平台，实现以下目标：

目标1：统一数据口径
- 关键结果：建立公司级指标字典
- 衡量标准：指标口径一致性达到95%以上

目标2：提升数据效率
- 关键结果：核心指标自动化更新
- 衡量标准：数据需求响应时间从5天降至1天内

目标3：支撑业务决策
- 关键结果：建立分层看板体系
- 衡量标准：周活跃用户覆盖80%以上的产品和运营人员

目标4：探索AI能力
- 关键结果：落地1-2个AI分析场景
- 衡量标准：完成POC验证并获得用户正向反馈

### 1.3 项目范围

本期包含（P0和P1）：
- 核心指标看板
- 指标口径管理
- 数据预警功能
- 基础报表导出

下期规划（P2）：
- 自定义报表
- AI智能分析
- 数据权限管理
- 移动端适配

## 二、用户分析

### 2.1 目标用户

用户角色1：产品经理

基本信息：负责产品功能迭代和用户体验优化

核心任务：
- 评估功能上线效果
- 分析用户行为路径
- 发现产品问题和机会点

数据需求：
- 功能渗透率和使用频次
- 新功能前后留存对比
- 用户反馈和评分趋势

使用频率：每日查看

痛点：
- 数据获取慢，影响决策速度
- 数据分散在多个地方，整合困难
- 想做对比分析但不会写SQL

用户角色2：运营人员

基本信息：负责用户增长和活动运营

核心任务：
- 监控运营指标达成情况
- 评估活动效果
- 制定运营策略

数据需求：
- DAU、新增、留存等核心指标
- 渠道质量分析
- 活动ROI分析

使用频率：每日查看，活动期间高频

痛点：
- 实时数据缺失，无法及时调整策略
- 报表更新滞后，周一才能看到上周数据
- 跨渠道数据对比困难

用户角色3：管理层

基本信息：部门负责人及以上

核心任务：
- 了解业务整体健康度
- 做出资源分配决策
- 向上汇报业务进展

数据需求：
- 核心指标概览
- 趋势变化和异常预警
- 竞品对比数据

使用频率：每周查看

痛点：
- 数据分散，需要多处查看
- 数据颗粒度太细，缺乏总结
- 环比同比信息不直观

### 2.2 用户旅程

以产品经理评估新功能效果为例：

阶段1：发现需求
- 触发点：新功能上线1周后
- 用户想法：我想知道用户用得怎么样
- 当前做法：找数据分析师提需求

阶段2：获取数据
- 等待时间：3-5天
- 用户情绪：焦虑，担心功能效果不好
- 当前做法：反复催促分析师

阶段3：分析数据
- 收到格式：Excel表格
- 用户动作：手动整理和分析
- 当前做法：自己做透视表和图表

阶段4：得出结论
- 用户动作：写分析报告
- 输出产物：PPT或文档
- 当前做法：大量时间花在数据整理而非分析

机会点：
- 在阶段2提供自助查询，无需等待
- 在阶段3提供可视化工具，减少手动操作
- 在阶段4提供AI摘要，提升总结效率

### 3.3 详细功能说明

#### 功能1：核心指标看板

功能描述：
以仪表盘形式展示产品核心指标，帮助用户快速了解业务整体状况。

用户故事：
- 作为产品经理，我希望打开平台就能看到今天的核心数据，以便快速了解产品状态
- 作为运营人员，我希望看到指标的涨跌变化，以便判断是否需要关注
- 作为管理层，我希望看到业务大盘概览，以便做出决策

功能要点：

指标卡片：
- 展示指标名称、当前值、环比变化
- 环比上涨显示绿色向上箭头
- 环比下跌显示红色向下箭头
- 支持点击查看指标详情

默认展示指标：
- DAU（日活跃用户数）
- 新增用户数
- 次日留存率
- 付费转化率（如适用）

数据更新：
- 数据每日自动更新，更新时间为凌晨2点
- 页面显示最后更新时间
- 支持手动刷新

交互流程：

用户打开平台
→ 系统加载默认看板
→ 展示4-6个核心指标卡片
→ 用户可点击某个指标
→ 展开显示该指标的趋势图和明细

异常情况处理：

数据加载失败：
- 显示"数据加载失败，请刷新重试"
- 提供刷新按钮

数据延迟未更新：
- 显示"数据更新中"状态
- 显示上次更新时间

筛选条件无数据：
- 显示"暂无数据"
- 建议调整筛选条件

#### 功能2：多维度筛选

功能描述：
支持用户按不同维度筛选数据，满足不同分析场景。

筛选维度：

时间范围：
- 预设选项：今日、昨日、近7日、近30日、本月、上月
- 自定义：支持选择任意起止日期
- 限制：最多查询90天数据

地区：
- 支持选择：全部、国内、海外
- 支持多选
- 默认：全部

渠道：
- 支持选择：全部、应用商店、官网、广告投放等
- 支持多选
- 默认：全部

版本：
- 支持选择近10个版本
- 支持多选
- 默认：全部

用户类型：
- 支持选择：全部、新用户、老用户、付费用户
- 单选
- 默认：全部

交互设计：

筛选器位置：页面顶部，横向排列

筛选器联动：选择条件后自动更新数据

重置功能：一键重置所有筛选条件

筛选记忆：记住用户上次的筛选条件

#### 功能3：数据预警

功能描述：
当核心指标出现异常波动时，自动发送告警通知。

预警规则：

规则类型1 - 阈值告警：
- 指标超过或低于设定阈值时触发
- 例如：崩溃率超过0.5%时告警

规则类型2 - 波动告警：
- 指标环比波动超过设定比例时触发
- 例如：DAU环比下降超过10%时告警

规则类型3 - 趋势告警：
- 指标连续多日同向变化时触发
- 例如：留存率连续3天下降时告警

通知方式：
- 站内消息：登录后显示告警列表
- 邮件：发送到用户注册邮箱
- 企业微信/钉钉：推送到群机器人

预警配置：
- 管理员可配置预警规则
- 用户可订阅关注的指标
- 支持设置免打扰时段

## 六、技术方案

### 6.1 技术架构

前端技术：
- 框架：React
- 图表库：ECharts 或 AntV
- UI组件：Ant Design
- 状态管理：Redux

后端技术：
- 框架：Spring Boot 或 Python Flask
- 数据库：MySQL（元数据）+ ClickHouse（分析数据）
- 缓存：Redis
- 任务调度：Airflow

数据层技术：
- 数据仓库：基于Hive或ClickHouse
- ETL工具：DataX 或 Spark
- 指标计算：dbt

### 6.2 数据流设计

数据采集：
- 客户端埋点数据通过Kafka实时上报
- 业务数据库通过CDC同步到数据仓库
- 第三方数据通过API定时拉取

数据加工：
- 原始数据落入ODS层
- 清洗后进入DWD明细层
- 聚合后进入DWS汇总层
- 指标数据进入ADS应用层

数据服务：
- 通过API向前端提供数据
- 预计算热门指标，存入Redis缓存
- 复杂查询走ClickHouse

### 6.3 性能要求

页面加载：首屏加载时间小于3秒

数据查询：常规查询响应时间小于2秒

数据更新：每日凌晨2点前完成T+1数据更新

并发支持：支持100人同时在线使用
